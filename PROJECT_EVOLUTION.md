## 1. 项目目标

Agent Orchestra 的最终目标是成为 OpenClaw 生态的**可视化调度中枢**：

- 实时掌握所有 Agent 状态、任务进度、系统健康
- 可派发、编排、监控多 Agent 协作任务
- 提供数据驱动的决策支持（趋势分析、负载洞察、智能推荐）
- 最终从「管理面板」演进为「可扩展的 Agent 平台」

## 4. 架构概述

```
项目结构：Node.js 原生 HTTP 单体应用
├── server.js          — 主入口，HTTP 路由 + 所有 API
├── lib/               — 业务模块（24 个文件）
│   ├── api-docs.js          — OpenAPI 3.0 文档生成 + Swagger UI
│   ├── runtime-utils.js     — OpenClaw API 交互
│   ├── workflows.js         — 工作流核心
│   ├── workflow-runner.js   — 工作流执行引擎
│   ├── notification-*.js    — 通知子系统（6 个文件）
│   ├── scheduled-backup*.js — 备份子系统（3 个文件）
│   ├── combination-*.js     — 组合推荐子系统
│   └── ...
├── public/            — 前端（单 HTML + app.js + styles.css）
├── data/              — JSON 文件持久化
├── Dockerfile         — 多阶段构建，Node.js 20 Alpine
├── docker-compose.yml — 一键部署配置
└── DEPLOY.md          — 部署文档
```

关键问题：server.js 路由过于集中（~6000行），lib/ 模块间耦合不明确，前端为单文件 SPA。

---

## 5. 已知问题与技术债

1. **server.js 臃肿** — 所有路由集中在一个文件（~6000行），应拆分为路由模块
2. **无测试覆盖** — 没有任何自动化测试，每次修改靠人工验证
3. **前端单文件** — public/app.js 承载了所有 UI 逻辑（~9500行），难以维护
4. **JSON 文件存储** — data/ 目录用 JSON 做持久化，并发安全和性能都有隐患

## 6. 演进路线图

### 近期（1-3 轮）— 产品化收敛
- **代码架构拆分**：server.js 路由模块化，按功能域拆分
- **自动化测试框架**：核心 API 端点的集成测试
- **API 验证中间件**：基于 OpenAPI spec 的请求/响应验证

### 中期（4-10 轮）— 平台化跃迁
- **插件系统**：允许通过插件扩展新面板、新通知渠道、新数据源
- **性能优化**：大数据量下的分页、懒加载、缓存
- **多用户认证**：真实的登录/鉴权系统
- **前端模块化**：拆分 app.js 为组件

### 远期愿景
- 成为 OpenClaw 的官方可视化管理面板
- 支持插件市场，社区可贡献面板组件
- 从「管理工具」变成「Agent 协作平台」
